# Linux Interview Prep — 50 Questions for a 3–5 Year DevOps Engineer

> Focus: not memorizing commands, but showing you can **troubleshoot production systems logically**. For every question: understand *why* it's asked, give the strong answer, know the commands, and have a production story ready.

---

## Section 1: Linux Basics & Daily Work

### 1. What happens internally when you run `ls`?
Shell looks up the binary via `PATH` → kernel creates a process → binary makes syscalls → kernel reads filesystem metadata → output returned to terminal.
**Flow:** `Shell → PATH lookup → Kernel creates process → Filesystem access → Output`
**Prod angle:** `ls` taking 15s instead of 0.1s → suspect NFS latency, millions of files, or filesystem corruption. Relevant to K8s volumes, NFS mounts, EBS performance.
**Tip:** Don't just say "it lists files" — walk through Shell → Kernel → Filesystem → Output.

### 2. Difference between `su` and `sudo`
- `su` — switches user entirely, needs target user's password.
- `sudo` — runs specific commands with elevated privileges, keeps audit logs, enforces least privilege.
```
su root
sudo systemctl restart nginx
```
**Prod angle:** Multiple engineers on prod — `su root` makes actions untraceable; `sudo` gives audit logs/compliance (SOC2, ISO27001).
**Tip:** Always mention the **Least Privilege Principle**.

### 3. Explain the Linux boot process
`BIOS/UEFI → GRUB → Kernel → systemd → Services → Login`
BIOS/UEFI initializes hardware → GRUB loads kernel → kernel mounts root filesystem → systemd starts services and prepares for user access.
**Prod angle:** A bad `/etc/fstab` entry can prevent boot — knowing the sequence tells you where it failed.

### 4. Purpose of `/proc`
A virtual filesystem generated by the kernel exposing real-time process, memory, CPU, and networking info.
```
cat /proc/cpuinfo
cat /proc/meminfo
cat /proc/self/status
cat /proc/PID/status   # investigate a specific process's memory
```
**Tip:** Say "virtual filesystem" explicitly.

### 5. How to check OS and kernel version
```
cat /etc/os-release   # OS details
uname -r               # kernel version
hostnamectl             # full host info
```
**Prod angle:** K8s node instability traced to a kernel networking bug — checking kernel version first prevents chasing the wrong root cause.

### 6. What is an inode and why does it matter?
Filesystem structure storing metadata — owner, permissions, size, timestamps, disk block locations. **Does NOT store the filename** (filename → inode → data blocks is a separate mapping).
```
df -i                        # inode usage
find /var -type f | wc -l    # find dirs creating excess files
```
**Prod angle (classic):** `df -h` shows 40% disk used (looks healthy) but apps can't create files/pods fail — because `df -i` shows inodes 100% used. Disk space ≠ available inodes.
**Tip:** "A filesystem can have free space but still fail due to inode exhaustion."

### 7. Hard link vs soft (symbolic) link
- **Hard link** → points directly to the inode; survives deletion of the original filename.
- **Soft link** → points to a file path; breaks if the target is removed.
```
ln file1 file2      # hard link
ln -s file1 file2   # soft link
```
**Prod angle:** Blue-green deployments — a symlink like `current-release → release-v11` allows instant cutover.
**Tip:** Hard Link → Inode. Soft Link → Path.

### 8. What happens if `/etc/fstab` has a wrong entry?
Filesystems may fail to mount at boot; severe cases → emergency mode or failed boot.
```
mount -a   # ALWAYS test fstab entries before rebooting
```
**Prod angle:** Adding a new EBS volume, updating fstab without testing, then rebooting → prod outage.
**Tip:** Always mention testing with `mount -a` before reboot.

### 9. How to find the largest files in a filesystem
`df -h` only shows filesystem-level usage, not which files/dirs are consuming space.
```
du -sh *
du -sh /* | sort -rh
find / -type f -exec du -h {} + | sort -rh | head
```
**Prod angle:** Disk 100% full → 300GB log file → root cause: log rotation failure.
**Tip:** Explain the drill-down: Filesystem usage → directory usage → file usage.

### 10. Runlevels / systemd targets
Traditional runlevels: `0=shutdown, 1=single-user, 3=multi-user, 5=graphical, 6=reboot`
Modern systemd targets: `multi-user.target`, `graphical.target`, `rescue.target`, `emergency.target`
```
systemctl get-default
systemctl isolate rescue.target
```
**Prod angle:** Server boots into `emergency.target` after filesystem failure — use rescue mode to recover.
**Tip:** Mention both old runlevels and new systemd targets to show breadth.

## Section 2: File Systems & Storage

### 11. ext4 vs XFS
- **ext4** — mature, stable, general-purpose workloads.
- **XFS** — built for scalability/performance, large files, enterprise workloads.
**Prod angle:** RHEL-based K8s clusters often use XFS because container workloads generate heavy filesystem activity.

### 12. What is LVM and why use it?
Abstraction layer between physical storage and filesystems enabling flexible resizing/allocation without major downtime.
`Physical Disk → Physical Volume → Volume Group → Logical Volume → Filesystem`
**Prod angle:** Storage hits 90% — instead of migrating data/rebuilding, just add a disk, extend LVM, resize filesystem.

### 13. Extending a logical volume without downtime
```
pvcreate /dev/sdb
vgextend vg_data /dev/sdb
lvextend -L +100G /dev/vg_data/lv_app
resize2fs /dev/vg_data/lv_app     # ext4
# OR
xfs_growfs /app                    # xfs
```
**Tip:** Many candidates forget the final filesystem resize step — mention it explicitly.

### 14. What happens when a filesystem hits 100%?
Apps fail to write logs, DBs may reject transactions, services crash, stability degrades.
`Filesystem full → logs fail → app errors → monitoring stops → outage`
```
df -h
du -sh *
```
**Prod angle:** ELK server — logs fill disk → Elasticsearch stops → observability itself is lost.
**Tip:** "Monitoring may fail before the application does" shows operational maturity.

### 15. Checking disk usage and inode usage
```
df -h   # storage
df -i   # inodes
```
Apps can fail even with free disk space if inodes are exhausted (see Q6).

### 16. `mount` vs `umount`
`mount` attaches a filesystem to the directory tree; `umount` safely detaches it.
```
mount /dev/sdb1 /data
umount /data
```
**Prod angle:** An attached EBS volume must be mounted before use.

### 17. What is a bind mount?
Maps one directory to another location within the same filesystem hierarchy.
```
mount --bind /app /backup/app
```
**Prod angle:** Docker/K8s map host directories into containers this way constantly.

### 18. Troubleshooting disk I/O slowness
```
iostat
iotop
sar
```
Workflow: `Application slow → check CPU → check memory → check disk I/O`
**Prod angle:** DB latency increase traced to EBS throughput saturation.

### 19. What is `fsck` and when to use it?
Checks/repairs filesystem inconsistencies — should generally run on **unmounted** filesystems.
```
fsck /dev/sdb1
```
**Prod angle:** Unexpected shutdown → filesystem corruption → fsck recovery.
**Tip:** Mention the risk of running fsck on a mounted filesystem.

### 20. Fields in `/etc/fstab`
`device / mount point / filesystem type / mount options / dump / fsck order`
```
/dev/sdb1 /data xfs defaults 0 0
```
Always validate with `mount -a` before reboot.

## Section 3: User Management & Security

### 21. `/etc/passwd` vs `/etc/shadow` vs `/etc/group`
- `/etc/passwd` — username, UID, GID, home dir, shell
- `/etc/shadow` — encrypted passwords, expiry/policy (restricted access)
- `/etc/group` — group names, GIDs, membership
```
cat /etc/passwd
sudo cat /etc/shadow
cat /etc/group
id username
```
**Prod angle:** Developer can't access `/var/log/app` → wrong group assignment → investigate with `id`.
**Tip:** "Authentication ≠ Authorization" — say this explicitly.

### 22. Linux permissions numerically
`Read=4, Write=2, Execute=1` — summed per owner/group/others.
```
755 → rwxr-xr-x
644 → rw-r--r--
chmod +x deploy.sh
```
**Prod angle:** Deployment fails with "Permission Denied" because the script is missing the execute bit.

### 23. What is the Sticky Bit?
Allows users to create files in a shared directory but prevents them deleting files owned by others.
Example: `/tmp` shows `drwxrwxrwt` — the trailing `t` = sticky bit.
**Prod angle:** Prevents User A from deleting User B's file in a shared dir.

### 24. `chmod` vs `chown` vs `chgrp`
```
chmod 755 app.sh              # permissions
chown ubuntu file.txt          # owner
chgrp developers file.txt      # group ownership
chown appuser:appgroup /data   # combined fix example
```

### 25. What is `umask`?
Defines which permission bits are stripped from newly created files/directories.
```
umask 022   # → files default to 644, dirs to 755
```
**Prod angle:** A bad umask setting creating files as `777` is a major security risk.

### 26. Locking/unlocking a user account
```
passwd -l username   # lock
passwd -u username   # unlock
```
**Prod angle:** Immediate account lock when an employee exits.

### 27. Giving `sudo` access safely
Follow Least Privilege, assign via controlled groups, ensure auditability/centralized logging.
Use `visudo` — **never** edit `/etc/sudoers` directly.
**Prod angle:** A dev accidentally gets full root access and deletes prod config → outage.

### 28. Root password lost — what do you do?
Boot into GRUB recovery → single-user/rescue mode → remount filesystem writable → reset root password → reboot.

### 29. What is SELinux and why is disabling it risky?
Provides Mandatory Access Control (MAC) — restricts process access even when standard Linux permissions would allow it.
**Prod angle:** A compromised web server has its blast radius limited by SELinux policy.
**Tip:** Never say "we just disable it" — interviewers dislike that answer.

### 30. Checking SELinux status/mode
```
sestatus
getenforce   # Enforcing / Permissive / Disabled
```
**Prod angle:** App can't access `/var/www/data` despite correct standard permissions — root cause is an SELinux policy violation.

## Section 4: Process, CPU & Memory

### 31. Process vs Thread
A process is an independent program with its own memory space; a thread is a lightweight execution unit within a process, sharing that process's memory.
**Prod angle:** A Java process spawns 200 threads and becomes slow — investigation focuses on thread usage, not process count.
**Tip:** "Threads share memory, processes do not."

### 32. Finding high CPU/memory processes
```
top / htop
ps aux --sort=-%cpu
ps aux --sort=-%mem
```
Workflow: `High CPU → identify process → check logs → check threads → root cause`
**Prod angle:** CPU at 98% → root cause: infinite loop in application code.

### 33. `nice` and `renice`
```
nice -n 10 backup.sh     # start with a given priority
renice 5 -p 1234          # change priority of running process
```
Priority scale: `-20` (highest) to `19` (lowest), `0` default.
**Prod angle:** Deprioritize a backup job on a DB server so the DB stays responsive.

### 34. What happens when the system runs out of memory?
RAM exhausted → swap used → performance drops → OOM Killer may terminate processes to protect stability.
```
free -m
vmstat
```
**Prod angle:** Java memory leak → RAM full → OOM Killer terminates the app.

### 35. RAM vs Swap
RAM = fast, volatile, primary memory. Swap = disk-based, much slower, backup memory used under pressure.
**Prod angle:** Swap increasing while RAM is at 100% — server stays alive but performance degrades.

### 36. Killing a hung process safely
```
ps aux              # identify
kill -15 PID        # SIGTERM: graceful
kill -9 PID         # SIGKILL: force, last resort
```
**Prod angle:** `kill -9` on a database process risks data corruption — always try SIGTERM first.

### 37. Load average — how to read it
```
uptime   # e.g. "1.20 1.40 1.60" = 1/5/15-min averages
```
Interpret **relative to CPU core count**: load 4 on a 4-core box = healthy; load 20 on 4 cores = problem.

### 38. Why does Linux "use" free RAM?
Linux aggressively uses spare RAM for filesystem caching — unused memory is wasted memory. Cached memory is reclaimed on demand.
```
free -m   # look at "available", not raw "free"
```
**Prod angle:** Seeing 95% memory used and panicking when the system is actually perfectly healthy.

### 39. What causes zombie processes?
A child process finishes but its parent never reaps its exit status.
```
ps aux | grep Z
```
**Prod angle:** A poorly written app accumulates thousands of zombies, filling the process table.

### 40. Troubleshooting high system load
Workflow: `High load → CPU → Memory → Disk I/O → Network → Application`
```
top          # CPU
free -m      # Memory
iostat       # Disk
sar -n DEV   # Network
```
**Prod angle:** Load average 40 but CPU only 20% — root cause is disk I/O wait, not CPU.

## Section 5: Networking & Services

### 41. Checking IP address and routing table
```
ip addr                 # interfaces
ip route                # routing table
ip route | grep default # default route
```
**Prod angle:** App can't reach the DB — missing default route means packets never leave the server.

### 42. TCP vs UDP (with examples)
- **TCP** — reliable, ordered, connection-oriented (acks/retransmits). *Examples: HTTPS, SSH, databases, email.*
- **UDP** — faster, connectionless, no delivery guarantee. *Examples: DNS, video streaming, VoIP, gaming.*
**Prod angle:** DNS outage traced to UDP port 53 being blocked — name resolution fails app-wide.

### 43. DNS and name resolution in Linux
`Application → Resolver → DNS server → IP address → Connection`
```
dig google.com
nslookup google.com
```
**Prod angle:** App can't reach `database.company.com` but direct IP works → DNS failure.

### 44. What controls local hostname resolution?
`/etc/hosts`, typically checked before DNS depending on resolver config.
```
cat /etc/hosts
```
**Prod angle:** Temporary `/etc/hosts` entry as a workaround when DNS is down.

### 45. Checking if a port is open/listening
```
ss -tulpn              # modern
netstat -tulpn          # legacy
ss -tulpn | grep 8080
```
Then test with `nc`, `curl`, or `telnet`.
**Prod angle:** App unavailable — port not listening because the service crashed.

### 46. Starting vs enabling a service
```
systemctl start nginx    # starts now (current session)
systemctl enable nginx   # starts automatically at boot
```
**Prod angle:** Server reboots, app is down — service was started manually before but never enabled.

### 47. Troubleshooting a service that won't start
Workflow: `Status → Logs → Configuration → Dependencies → Permissions`
```
systemctl status nginx
journalctl -xe
```
**Prod angle:** Nginx fails to start — logs show "port 80 already in use."

### 48. Firewall tools in Linux
`firewalld`, `iptables`, `nftables`, `ufw`, cloud Security Groups.
```
firewall-cmd --list-all
```
**Prod angle:** App unreachable — port 443 blocked by firewall policy.
**Tip:** Never say "I just disable the firewall."

### 49. Testing connectivity without ping (ICMP often blocked)
```
curl        # application-level test
nc / telnet # port connectivity
traceroute  # network path
```
**Prod angle:** Ping blocked, but `curl` confirms the service is actually healthy.

### 50. Server works with IP but not hostname — how do you fix it?
Investigation flow: `Hostname fails → check DNS → check resolver → check /etc/hosts → check firewall → verify DNS server`
```
cat /etc/resolv.conf
dig hostname
cat /etc/hosts
```
**Prod angle:** Classic root cause — `/etc/resolv.conf` points to an incorrect/unreachable DNS server.

## What interviewers are really testing (across all 50)
Not "do you know every command" but:
- Can you **think logically** and troubleshoot systematically?
- Can you stay calm during an incident?
- Can you identify root causes, not just symptoms?
- Can you explain your reasoning out loud?

Production systems fail with **symptoms**, not helpful error messages — the strongest engineers are the ones who know what to investigate next, not the ones who've memorized the most flags.
